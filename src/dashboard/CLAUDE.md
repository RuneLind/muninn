## Route Organization

Routes are split by domain in `routes/`:

| File | Endpoints |
|---|---|
| `data-routes.ts` | Main CRUD: messages, memories, goals, tasks, watchers, users, threads, connectors |
| `sse-routes.ts` | SSE streams on `/api/events`: `activity`, `stats`, `agent_status` (phase), `request_progress` (single-pane waterfall), and `agent_runs` (full snapshot of ALL AgentRuns for `/agents` — initial via `agentStatus.getAll()`, live via `subscribeAll`, throttled ~1/s in the tracker, tools capped at 20 per run) |
| `agents-routes.ts` | `/agents` live-agent dashboard page + `GET /api/agents/overview` (JSON: `running` + `upNext` + `recent`). Assembly is the pure, injectable `src/dashboard/agents-overview.ts` (`assembleAgentsOverview`). `running` from the AgentRun registry; `upNext` from `scheduled_tasks.next_run_at` + watchers (next fire mirrors `isScheduledTimeDue` — time-of-day slot dominates a naive `last_run_at + interval`; `computeWatcherNextRun`); `recent` is a four-source UNION keyed by span NAME (chat `%_message`/`telegram_voice` roots + `watcher:%` child spans via `src/db/agent-activity.ts`, extractor `haiku_usage` rows, and the registry completed-runs ring for gardener_drain/capture/research/per-task only — per-kind source-of-truth prevents double-counting). The payload also carries `estimates` — per-live-run `expectedDurationMs` keyed by `estimateIdentity(kind, name)` (`src/dashboard/agent-eta.ts`; watcher identities sourced from durable `watcher:<type>` span durations via `getWatcherRunDurations`, every other kind from the in-memory completed-runs ring, chat NEVER estimated, median of the last ~20) — and `processStartedAt` (the "history resets on restart" empty-state). The SSE `agent_runs` cards carry no estimate; the `/agents` client looks each live run up in `estimates` and computes pace (`elapsed/done × total`) locally where a gardener drain reports discrete progress (pace beats the history median). Never 5xx — degraded sources land in `errors[]` (a degraded `watcher_durations` just drops watcher estimates → elapsed-only). |
| `traces-routes.ts` | Trace spans: list, detail, waterfall data, cleanup |
| `search-routes.ts` | Knowledge base document search and management |
| `research-routes.ts` | `/research` page: cited Q&A over the shelf corpus (`GET /api/research/ask` SSE — retrieve via `researchKnowledge`, synthesize one cited answer; see `src/research/{corpus,answer,ask}.ts`) + the browse `/api/research/*` endpoints. Also the Jira research flow (`/api/research/chat`: trigger analysis, post to chat). |

| `graph-routes.ts` | Knowledge graph page + wikilink edge extraction across knowledge collections |

**Native `.mdx` pages (first-class markdown):** `src/wiki/store.ts` discovers `**/*.{md,mdx,html}`. A `.mdx` file is treated exactly like a `.md` page — frontmatter (title/tags/type), `[[wikilinks]]` + relative-link graph membership (wikilinks inside component bodies count), backlinks, and index-coverage (`isIndexableId` accepts `.mdx`; it counts as markdown, not `htmlPages`). It renders **inline** through `renderWikiHtml` → `formatWebHtml` (shared block AST, so `<Callout>`/`<Verdict>`/etc. render as components), never in the explainer iframe (its type is never `explainer`). Same-stem collisions resolve by precedence `.md` > `.mdx` > `.html` — the loser is dropped from the index (logged at warn), the read-side counterpart to mimir's authoring-side `scripts/mdx-explainer/checks.ts` guard. **Compile-pipeline SOURCE exclusion (load-bearing):** because the collision key is the BARE stem across all folders, a mimir MDX source at `blogs/src/<slug>.mdx` would otherwise SHADOW its own compiled explainer `blogs/<slug>.html` (`.mdx` rank 1 beats `.html` rank 2), silently dropping the explainer the reader is meant to serve and stripping its backlinks. So the store excludes a `.mdx` from discovery entirely when it is a pipeline source — detected surgically (`pipelineSourceSiblingHtml`) by the documented layout (immediate parent dir `src`) **reinforced by** a compiled sibling `<dir>/<stem>.html` actually existing and carrying the generated marker (`<!-- generated from`). A stray/hand-authored `.mdx` — even one under a `src/` dir with no compiled twin — stays a first-class native page; the exclusion is logged at DEBUG (expected pipeline shape), not the WARN used for real shadow collisions. muninn has no mermaid renderer, so a ```mermaid fence in a native page renders as a plain code block (compile such diagram pages to `.html` explainers via the mimir MDX pipeline instead). `src/wiki/similar.ts` strips component tags before building the semantic-search query so JSX-ish tokens don't pollute the embedding.: the reader's "Recently updated" sort key is `pageTimeMs` = the LARGER of mtime and the frontmatter `updated`/`created` date, so wikis that carry no frontmatter (mimir, melosys-kode-wiki — 216 of mimir's 220 pages) rank honestly instead of sinking to the bottom as undated, while a re-checked-out wiki (mtime reset) still keeps its frontmatter dates. A page is `excludedByRule` only when EVERY collection would never index it AND it isn't in the indexed union; an indexed page stays `indexed` whatever the rules say. **Graceful degrade:** an older huginn (no pattern fields) or an unreachable `/api/collections` ⇒ no partition, meta pages fall back into `missing` (the pre-#243 behavior). Membership compares on a normalized key (`normalizeRelPath` + Unicode **NFC** — the NFC fold stops macOS-NFD vs huginn-NFC Norwegian names showing as false missing+ghost pairs). Counting is **per-file** (not per-unique-key) so a case-only-distinct `Foo.md`+`foo.md` pair both count and the invariant `indexed + missing.length + excludedByRule.length === totalMd` holds. ## View System

All pages are server-side rendered HTML via template literals. No build step, no framework.

### Component Pattern

Each component in `views/components/` exports up to three functions:

- `*Styles()` — returns a CSS string (injected into `<style>`)
- `*Html()` — returns an HTML string (injected into `<body>`)
- `*Script()` — returns a JavaScript string (injected into `<script>`)

Pages (`views/*.ts`) compose components by calling all three and concatenating into a full HTML document.

### Shared Infrastructure

- `views/shared-styles.ts` — CSS variables (colors, spacing), base reset, header, nav. Imported by all pages.
- `views/components/helpers.ts` — Inline JS utilities: `esc()` (HTML escaping), `extractToolInputLabel()`, formatting helpers.
- `views/components/agent-presence.ts` — the **agent presence chip** (mounted on `/wiki/gardener`, `/summaries`, `/wiki`). A tiny, embeddable "is a matching agent live (or about to be)?" indicator: standalone 15s poll of `GET /api/agents/overview` (NO second SSE — the host pages own their own loops), filtered by `{ kinds?, bot? }`, deep-linking to `/agents`. Three states — `running` (pulsing `.pulse-dot` + "<Kind>: <phase> <done>/<total>" label + optional "~Nm left" ETA from the overview `estimates` map), `preflight` (amber "<Kind> starts in Nm" from a matching `upNext` within `PRESENCE_HORIZON_MS` = 60 min), and `none` (renders NOTHING). The pure model builder `computePresence` is DOM-free + unit-tested; the client JS in `agentPresenceScript` hand-mirrors it (keep in sync, like the `computeCardEta` mirror). Each mount is a one-liner styles+html+script composition.

### Pages

`page.ts` (main dashboard), `traces-page.ts`, `search-page.ts`, `search-document-page.ts`, `research-page.ts`, `logs-page.ts`, `mcp-debug-page.ts`, `serena-page.ts`, `summaries-page.ts` (unified YouTube + X-article summaries; composes the `sum-*` components, injects the `SOURCES` registry), `models-page.ts` (models overview — three grouped tables: Bots · Role assignments · Pipeline jobs; client-fetches `/api/models/overview`, bot selector re-scopes only the per-bot Pipeline rows), `agents-page.ts` (unified live-agent dashboard — Running zone re-renders on the `agent_runs` SSE event + rAF-ticks elapsed timers/bar widths [two-tier pattern from `request-progress-ui.ts`]; Up next + Recently finished from `/api/agents/overview`; reuses `traces-prompt-modal.ts` for prompt inspection and the bot selector for filtering. Recently-finished has a **Cost** column rendered via `fmtCost` — `$0.00` for a real 0, a dash for unknown; see the observability `CLAUDE.md` `costUsd` note).

## Real-Time Updates

### SSE Endpoints (sse-routes.ts)

- `/api/events` — activity feed events (message_in, message_out, memory, error, etc.), plus `agent_status` (phase), `request_progress` (single-pane waterfall), and `agent_runs` (full AgentRun snapshot for `/agents`, throttled ~1/s)

### Agent Status (`src/observability/agent-status.ts`)

Tracks in-flight requests in a `Map<requestId, RequestProgress>` (concurrent requests no longer clobber a shared slot). The read side still surfaces a single primary request, so the SSE `/request-progress` contract and the waterfall UI are unchanged. See `src/observability/CLAUDE.md` for the full method list and the request-scoped-mutator rule.

1. `startRequest()` — creates RequestProgress keyed by a fresh requestId
2. `toolStart(requestId, …)`/`toolEnd(requestId, …)` — records tool calls with timing
3. `completeRequest(requestId, …)` — marks done, auto-clears that request after 30s
4. Phases: `idle`, `receiving`, `transcribing`, `building_prompt`, `calling_claude`, `saving_response`, `sending_telegram`, `sending_slack`, `synthesizing_voice`, `running_task`, `checking_goals`, `running_watcher`

### Activity Log (`src/observability/activity-log.ts`)

- In-memory ring buffer (max 500 events) + fire-and-forget DB persistence
- `loadFromDb()` called after DB init to hydrate on startup
- `stats` getter: messages today, avg response time, total cost

## MCP Debug Client (`src/ai/mcp-tool-caller.ts`)

- Supports both stdio (spawn process) and HTTP (StreamableHTTP) transports
- Connection pool keyed by `bot:server` — reuses existing connections
- Timeouts: 30s connect, 120s tool call
- Used by the MCP debug page (`tools-routes.ts`), Serena management, and the `openai-compat` connector

## Common Pitfalls

1. **SSE format**: Events must be `data: JSON\n\n` — missing double newline breaks the stream.
2. **Agent status waterfall is single-pane**: the data layer tracks all concurrent requests (per-`requestId` Map), but `/request-progress` emits only the primary (most-recently-started) request, so the overlay shows one at a time.
3. **Component pattern**: Always export all three (*Styles, *Html, *Script) even if empty — pages expect them.
4. **MCP connection cleanup**: Always call `disconnectServer()` or `disconnectAll()` — leaked stdio processes stay alive.
