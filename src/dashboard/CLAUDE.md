The Ask route also passes a per-wiki `systemPrompt` (`buildSynthesisSystemPrompt(...)` framed to the wiki name) through the shared `research-sse.ts` helper; `/research` leaves it unset (byte-identical LC prompt). Switching the synthesis bot also switches the retrieval identity (decomposer backend + `research_citations`/`search_signals` attribution). The right-column Ask tab holds only the controls (question box · status line · clickable session-history list); the **answer renders in the main article pane** as a formatted article. After the terminal `done`, the route emits a trailing `answer_html` event — the final answer markdown rendered through the shared wiki pipeline (`renderAskAnswerHtml` in `src/wiki/ask-render.ts` → `formatWebHtml`) with in-range `[n]` markers linkified to their matched pages. The client renders the streaming buffer progressively through the same `formatWebHtml` markdown pipeline (throttled to one animation frame per burst; the import-safe helper lives in `views/components/wiki-ask-render.ts` and is unit-tested), so headings/lists/code grow formatted during the stream, then swaps in the final rendered HTML on `answer_html` (falls back to the progressively-formatted text if it never arrives). The final-render hook is opt-in on the shared `research-sse.ts` helper (`renderAnswerHtml`); only the wiki route enables it (`/research` renders client-side). `/api/wiki/digest` is the start view's **What's new** card: `generateWikiDigest` (`src/wiki/digest.ts`) reads the wiki's `log.md`, summarizes the recent window (≤14 days / ≤30 entries / ≤15 KB) via one `executeOneShot` call on the wiki's synthesis bot (owner-routed via `resolveWikiSynthesisBot`, same as Ask), and marks resolvable page mentions as `[[wikilinks]]`. Cached per wiki in an in-memory Map keyed by name (mtime-validated via `digestCacheDecision`; `?refresh=1` bypasses); a wiki with no `log.md` returns `{ digest: null }`. Stored bullets stay plain markdown — the route renders them to reader HTML at response time. The `/wiki` page also annotates each picker option with the wiki's freshness date (`log.md` mtime date, fallback newest page date). `GET /api/wiki/index-coverage?wiki=` is the start view's read-only **Index** card: it compares the wiki's on-disk `.md` pages against the deduped UNION of its backing `collections`' huginn doc ids (huginn doc `id` IS the wiki-relative path) and reports `missing` (actionable `.md` gaps — pages a collection SHOULD index but doesn't), `excludedByRule` (`.md` pages no collection would ever index by design — the meta denylist `index.md`/`log.md`/`CLAUDE.md`/`plans/*`/dotfiles, or out-of-scope of a scoped `^life/.*` include; informational, NOT missing), `ghosts` (indexed ids with no file — restricted to `.md`/`.html` ids so mimir's indexed `.gitignore`/`.txt`/`.json` don't show as false ghosts), `htmlPages` (explainers — informational, never counted missing), plus `indexed`/`totalMd`. **Pattern sourcing:** the exclude/include rules come from huginn's `GET /api/collections` (`includePatterns`/`excludePatterns` per collection — one degrade-tolerant call, **no hardcoded pattern copies in muninn**); the pure layer mirrors huginn's `FilesDocumentReader` semantics — BOTH pattern sets applied with Python `re.fullmatch` (anchored `^(?:pat)$`), matched against the posix wiki-relative path in ORIGINAL case (huginn regexes are case-sensitive). A page is `excludedByRule` only when EVERY collection would never index it AND it isn't in the indexed union; an indexed page stays `indexed` whatever the rules say. **Graceful degrade:** an older huginn (no pattern fields) or an unreachable `/api/collections` ⇒ no partition, meta pages fall back into `missing` (the pre-#243 behavior). Membership compares on a normalized key (`normalizeRelPath` + Unicode **NFC** — the NFC fold stops macOS-NFD vs huginn-NFC Norwegian names showing as false missing+ghost pairs). Counting is **per-file** (not per-unique-key) so a case-only-distinct `Foo.md`+`foo.md` pair both count and the invariant `indexed + missing.length + excludedByRule.length === totalMd` holds. Pure logic + the degrade contract live in `src/wiki/index-coverage.ts` (`computeIndexCoverage`, `buildIndexCoverageResponse`); the route sources listings via the shared `listCollections` helper (`src/summaries/list-collections.ts`, also used by the summaries stats/backlog paths). Never-5xx: a wiki with no `collections` (or unknown / dir-missing) returns 200 + `{ error }` with null coverage fields; a failed collection listing returns 200 + `errors[]` AND **suppresses** the coverage fields (`totalMd`/`indexed`/`missing`/`excludedByRule`/`ghosts` → null — a partial union would flag really-indexed pages as missing), `htmlPages` stays. `?refresh=1` busts the page-index TTL cache only (listings aren't muninn-side cached in v1). The card (client `wiki-browser.ts`, styles in `wiki-page.ts`) loads lazily on the start view next to the What's-new card (fetch-coalesced like the digest — a manual refresh racing the lazy load can't double-fetch); missing/meta pages link into the reader, missing/meta/ghost lists expand on click; a degraded/failed fetch shows a quiet "index status unavailable" line that still surfaces the preserved informational counts (htmlPages), a no-collections wiki hides the card. **Manual reindex (PR):** the Index card carries a **[Reindex now]** button backed by `POST /api/wiki/reindex?wiki=` (fans huginn's `POST /api/collections/<c>/update` over each backing collection) + `GET /api/wiki/reindex-status?wiki=` (proxies each `/api/collections/<c>/update-status`). Same registry resolution + `?bot=` alias + clean no-collections error as index-coverage. huginn's CAS **409** (a rebuild — nightly job or a prior trigger — already in flight) maps to the honest `already-running` state, **not** a failure; an unreachable huginn ⇒ 200 with per-collection `error` entries; a failed status fetch ⇒ `status: "unknown"` + error for that entry. **Never 5xx** and **no muninn-side mutex** — huginn's CAS is the single serialization point. The client renders per-collection state inline, then polls status every 3 s until no collection is `running` (gives up quietly after 3 consecutive poll failures, survives the card disappearing mid-poll), disables the button for the cycle, and on settle re-fetches `index-coverage?refresh=1` so missing/ghosts reflect the fresh index; a terminal `failed` status renders huginn's error text on the card (first visibility into a silently failing nightly job). Response-assembly (409→`already-running` mapping, unreachable→error entries, multi-collection fan-out, status→`unknown` degrade) is the pure/injectable `src/wiki/reindex.ts` (`buildReindexResponse`/`buildReindexStatusResponse`); the route implements the HTTP posters via `fetchKnowledgeApi` (reading `KnowledgeApiError.upstreamStatus` to detect the 409). |

The run's work fn owns the progress-map writes + the `cancelRequested` read and threads them into `runGardener` as hooks (`onProgress`/`shouldAbort`/`onAborted`); the route's closure only forwards them. `POST /api/wiki/gardener/backlog-cancel` soft-cancels the in-flight drain (`requestBacklogCancel` flips the run's cancel flag; the runner stops after ≤1 more draft, keeps every persisted proposal, returns the undrafted clusters' docs to the offered set); no run in flight ⇒ no-op 200 `{cancelled:false}` (same resolution/watcher-seeded guards as reset). `POST /api/wiki/gardener/backlog-reset` clears the offered-memory snapshot. **Crash safety (PR 3):** the drain persists a **run journal** to `watcher_snapshots` key `backlog:run` (`{startedAt, batchKeys}`) BEFORE offering the batch, and the last-run outcome to `backlog:lastRun` (durable fallback the GET reads after a restart drops the in-memory map). A fresh Ingest **auto-recovers a pending journal in-mutex as its first work-fn step** (never a route pre-flight — that's a TOCTOU) so a click before Recover/Dismiss can't orphan the strand. ## View System

All pages are server-side rendered HTML via template literals. No build step, no framework.

### Component Pattern

Each component in `views/components/` exports up to three functions:

- `*Styles()` — returns a CSS string (injected into `<style>`)
- `*Html()` — returns an HTML string (injected into `<body>`)
- `*Script()` — returns a JavaScript string (injected into `<script>`)

Pages (`views/*.ts`) compose components by calling all three and concatenating into a full HTML document.

### Shared Infrastructure

- `views/shared-styles.ts` — CSS variables (colors, spacing), base reset, header, nav. Imported by all pages.
- `views/components/helpers.ts` — Inline JS utilities: `esc()` (HTML escaping), `extractToolInputLabel()`, formatting helpers.

### Pages

`page.ts` (main dashboard), `traces-page.ts`, `search-page.ts`, `search-document-page.ts`, `research-page.ts`, `logs-page.ts`, `mcp-debug-page.ts`, `serena-page.ts`, `summaries-page.ts` (unified YouTube + X-article summaries; composes the `sum-*` components, injects the `SOURCES` registry), `models-page.ts` (models overview — three grouped tables: Bots · Role assignments · Pipeline jobs; client-fetches `/api/models/overview`, bot selector re-scopes only the per-bot Pipeline rows), `agents-page.ts` (unified live-agent dashboard — Running zone re-renders on the `agent_runs` SSE event + rAF-ticks elapsed timers/bar widths [two-tier pattern from `request-progress-ui.ts`]; Up next + Recently finished from `/api/agents/overview`; reuses `traces-prompt-modal.ts` for prompt inspection and the bot selector for filtering).

## Real-Time Updates

### SSE Endpoints (sse-routes.ts)

- `/api/events` — activity feed events (message_in, message_out, memory, error, etc.), plus `agent_status` (phase), `request_progress` (single-pane waterfall), and `agent_runs` (full AgentRun snapshot for `/agents`, throttled ~1/s)

### Agent Status (`src/observability/agent-status.ts`)

Tracks in-flight requests in a `Map<requestId, RequestProgress>` (concurrent requests no longer clobber a shared slot). The read side still surfaces a single primary request, so the SSE `/request-progress` contract and the waterfall UI are unchanged. See `src/observability/CLAUDE.md` for the full method list and the request-scoped-mutator rule.

1. `startRequest()` — creates RequestProgress keyed by a fresh requestId
2. `toolStart(requestId, …)`/`toolEnd(requestId, …)` — records tool calls with timing
3. `completeRequest(requestId, …)` — marks done, auto-clears that request after 30s
4. Phases: `idle`, `receiving`, `transcribing`, `building_prompt`, `calling_claude`, `saving_response`, `sending_telegram`, `sending_slack`, `synthesizing_voice`, `running_task`, `checking_goals`, `running_watcher`

### Activity Log (`src/observability/activity-log.ts`)

- In-memory ring buffer (max 500 events) + fire-and-forget DB persistence
- `loadFromDb()` called after DB init to hydrate on startup
- `stats` getter: messages today, avg response time, total cost

## MCP Debug Client (`src/ai/mcp-tool-caller.ts`)

- Supports both stdio (spawn process) and HTTP (StreamableHTTP) transports
- Connection pool keyed by `bot:server` — reuses existing connections
- Timeouts: 30s connect, 120s tool call
- Used by the MCP debug page (`tools-routes.ts`), Serena management, and the `openai-compat` connector

## Common Pitfalls

1. **SSE format**: Events must be `data: JSON\n\n` — missing double newline breaks the stream.
2. **Agent status waterfall is single-pane**: the data layer tracks all concurrent requests (per-`requestId` Map), but `/request-progress` emits only the primary (most-recently-started) request, so the overlay shows one at a time.
3. **Component pattern**: Always export all three (*Styles, *Html, *Script) even if empty — pages expect them.
4. **MCP connection cleanup**: Always call `disconnectServer()` or `disconnectAll()` — leaked stdio processes stay alive.
